[[RSG]] [[20190915]] [[gztool]] [[chunk]] :<#acceder a chunks aislados de un gzip> 


<https://unix.stackexchange.com/questions/429197/reading-partially-downloaded-gzip-with-an-offset/541903#541903> 
++++++++++++++++++++++++
Just FWIW, **gzip can be accessed randomly, if a previous *index* file has been created...**

I've developed a command line tool that can quickly and (almost-)randomly access access a gzip if an index is provided (if it is not provided, it is automatically created):

https://github.com/circulosmeos/gztool

I think this could be used to access chunks of the original gzip file, if those chunks are retrieved at the specific byte points the index has pointed to. Later data extraction would require to fill zeroes, which can be done with *sparse data* to which the chunk would be appended:

For example if an index points starts at byte compressed byte 1508611 of the gzip file and we want 1M compressed bytes after that:

    $ dd if=/dev/zero of=chunk.gz seek=1508610 bs=1 count=0
    $ curl -r 1508611-2508611 https://example.com/db/backups/db.sql.gz >> chunk.gz

Note that `chunk.gz` will occupy on disk only the chunck size!

Now the complete (previously only-once created) index must also be retrieved. Note that indexes are ~0.3% of gzip size.

    $ curl https://example.com/db/backups/db.sql.gzi -o chunk.gzi

And now the extraction can be done with [gztool][1]. The corresponding uncompressed byte of compressed-1508611 must be known, but the index can show this info with `gztool -ll`. Let's suppose it is byte 9009009:

    $ gztool -b 9009009 chunk.gz > extracted_chunk.sql

`gztool` will stop when the `chunk.gz` file ends.

Maybe tricky, but would run without changing compression method nor already compressed files. But indexes would need to be created for them.

  [1]: https://github.com/circulosmeos/gztool
